## Engineering principles

These apply across every module. Some areas already follow them tightly; others are evolving. The bar isn't perfection ŌĆö every change should leave the surrounding code more maintainable than it found it. Small decreases in code health compound into rewrites; small improvements compound into a codebase a new engineer can join in a week.

- **Modules align with concerns, not technical layers ŌĆö Single Responsibility Principle at module scale.** The Single Responsibility Principle (Robert C. Martin) says a unit of code should have one reason to change. At module granularity: each module should answer one question, nameable in a single sentence. If you can't, it's drifting toward a junk drawer ŌĆö split the concerns out, or fold the module into its real owner. Boundaries follow what changes together, not generic "models / views / controllers" buckets.
- **Public surface is small and explicit.** A package's entry point is its contract; leading underscores on internal modules and subpackages signal "don't import from here". Explicit re-export lists pin what consumers may actually depend on. The smaller the public surface, the cheaper internal refactors become.
- **Deterministic public entry points; subpackages organize by concern, not by technical layer.** Each package exposes one canonical surface (its `__init__.py` re-export list); every other module under it is *internal* even when not underscore-prefixed ŌĆö `grove.core.contracts.branch_plan` is plumbing, `grove.core` is API, and the distinction itself is the contract. External callers stick to the package root; internal refactors move freely below. Within a package, split by concern (`contracts/` for wire-level Pydantic shapes that cross client boundaries, `git.py` for the git side-effect surface, `manager.py` for orchestration, `tmux.py` for the tmux side-effect surface), **never** by technical layer ŌĆö no generic `helpers/`, `utils/`, `models/`, `services/` directories. Those are junk drawers in disguise: every junior engineer drops another file in and the boundary erodes. When one file accumulates two answers to two questions, split it; three small files with one job each beats one big file with three. The directory tree should be a map a developer can navigate without grep ŌĆö `where does the discriminated union live?` ŌåÆ `contracts/branch_plan.py`, no search required. This is also the backstop for the YAGNI / KISS / code-as-poem rules: once boundaries are explicit, "where does this new responsibility belong?" has a deterministic answer, which removes the pressure to either over-abstract (build a generic framework on speculation) or under-abstract (drop another helper into the junk drawer because no shape fits).
- **Dependencies flow inward.** Orchestration layers import from utility layers; the reverse is a smell. When a low-level helper has to know about a high-level caller, the boundary is wrong ŌĆö most circular-import pain traces back to this.
- **Boring code beats clever code.** Reuse the pattern already established in the project. Local cleverness has a price every reader pays; if you must deviate, name the reason inline. Predictability across modules is what keeps onboarding cheap as the team grows.
- **Add structure only when there's a real concern to separate ŌĆö You Aren't Gonna Need It (YAGNI).** YAGNI, from Extreme Programming, says: build only what the current requirement demands; do not add abstraction on speculation. Three similar lines is fine. A new helper, class, or subpackage costs review surface for years ŌĆö pay only when one engineer can plausibly own the new boundary, not for hypothetical future scaling. Applies to abstraction layers as much as to features.
- **Strong types where they catch bugs.** Narrow literal types for string sets that drive branching, structured types for conditional payload schemas, class-level constants marked as such, explicit return types everywhere. Escape-hatch types (the language's `Any` equivalent) only for genuinely heterogeneous external data, narrowed at the boundary. Introduce protocols / interfaces only when more than one real implementation exists.
- **Pydantic at public-contract boundaries; plain dataclass for in-process state.** Sharper form of the existing config/state rule: Pydantic for *anything that crosses a client/server boundary now or could in the future* ŌĆö config, request and response payloads, discriminated unions of intent. Plain `@dataclass(slots=True)` for internal mutable state (`WorkspaceState`) and intermediate IR that never leaves the engine (`ResolvedBranch`, `WorkspaceEvent`). Discriminated unions over Pydantic v2 use a `kind: Literal["..."]` tag and `Annotated[Union, Field(discriminator="kind")]` ŌĆö that's what gives every future client (web form, JSON-RPC) a generated JSON Schema for free, with `extra="forbid"` catching typos at the boundary. The `grove/core/contracts/` submodule is the canonical home for these wire-level shapes; never import them from a TUI module ŌĆö they belong to the engine and travel outward. **When in doubt, ask: would a non-Python client ever construct or receive this?** If yes, Pydantic. If no, dataclass.
- **Class-encapsulated atomic state ŌĆö code as poem, not a junk drawer of free helpers.** State and the methods that operate on it live together as a single class. Prefer instance / `@classmethod` / `@staticmethod` on the right type over root-level free functions, especially private ones (`_with_status`, `_init_outcome`, `_replace`, `_make_branch_name`). Free helpers in module scope are usually a missing class ŌĆö either the type that owns the state they manipulate, or a small named helper class for that concern. Use `@staticmethod` when the operation belongs to a type but doesn't need `self`; use `@classmethod` for alternate constructors (e.g., `GitRepo.detect(cwd)`). Don't go on a global crusade ŌĆö when you touch a module, leave its class structure a little tighter than you found it: convert one or two of its free helpers into proper members, fold a literal collection into a class attribute, give an anonymous tuple of fields a name. Tests pin behavior, not implementation, so this kind of inward refactor is safe to do as you pass through. Three signs you're paying the right amount of attention: (1) every state field is on a class, attribute-named to describe what it represents; (2) docstrings on the class and on every public method explain *why* the type exists, not what each line does; (3) the public surface of the package fits in one screen ŌĆö internal helpers are private to the file or inside the class, not re-exported.
- **Side effects at the edges, pure logic in the middle.** Input/output, network, database sessions, and time-of-day belong at the module boundary ŌĆö request handlers, fetchers, drivers. The decision logic in between should be testable without those. Best-effort side effects must isolate their failures: bounded timeout, structured log per outcome, never re-raise into the caller's retry path.
- **Tests pin contracts, not implementation.** When a test patches a private symbol, that path becomes an implicit contract ŌĆö moving it silently no-ops the patch and the test still passes. Either surface the seam publicly or update the test in the same commit. Hidden coupling between test and implementation is the most common cause of refactor friction.
- **Comments and docstrings explain WHY, not WHAT.** Names and types document the what; comments and docstrings carry the constraint, the trade-off, the past incident, the surprising invariant. Docstrings render in Integrated Development Environment (IDE) hover ŌĆö write the first line for the engineer deciding whether to call this, then add Args / Returns / Raises only for things the type signature doesn't already convey.
- **Keep It Simple, Stupid (KISS) and Don't Repeat Yourself (DRY) are the core philosophy.** Bias toward less code, not more. Before writing anything custom, search for an existing library or an existing utility in the codebase. **Pattern: proven library for infrastructure, custom code only for business logic.**
- **Mechanism, not policy ŌĆö configuration cascades at the consumption surface.** This rule applies to the *user-facing* surface of the project ŌĆö the framework, prompt, model, and workflow choices a developer using this software forms opinions about ŌĆö not to the internal architecture. From the Unix philosophy: provide mechanism, leave policy to the user. Defaults are sensible but held lightly, and any value a developer might reasonably want to change is reachable from outside the code. Configuration layers by specificity ŌĆö built-in ŌåÆ project ŌåÆ team ŌåÆ machine ŌåÆ user ŌåÆ invocation ŌĆö each layer overriding the previous, so a team can pin a shared standard while an 
  individual still owns the last word on their own setup. This affordance is only sustainable because the internal codebase stays strict on Don't Repeat Yourself (DRY) and Keep It Simple, Stupid (KISS): one boring implementation of each capability inside, with the override cascade resolving on top.

### Other rules

- Code validates itself at the point of definition (schema validators, strict configs that forbid unknown fields).
- Define logic once; call everywhere ŌĆö when a rule is reused by more than one caller, it lives in one place.
- Smallest diff that solves the problem. No speculative abstractions.
- Keep published contracts stable: interface names, method signatures, and field names don't move under consumers without coordination.
- Tests prefer real code paths; stub only Input/Output (I/O) boundaries. Cover full orchestration loops with in-memory fakes for external services and model outputs.
- Type hints stay precise; avoid escape-hatch types.
- Gitmoji plus Conventional Commits for commit messages (e.g. `Ō£© feat(*optional-scope1,*scope2): ...`).
- Do not push unless explicitly asked.
- Treat Large Language Models (LLMs) and agents as non-deterministic black-box Application Programming Interfaces (APIs); avoid anthropomorphic language.

## Key Session lessons (non-trivial)

Engine + cross-cutting only. TUI-specific lessons live in [`src/grove/tui/CLAUDE.md`](src/grove/tui/CLAUDE.md); webapp-specific lessons live in [`webapp/CLAUDE.md`](webapp/CLAUDE.md).

- Ask users about any assumptions or thoughts about the project to ensure that we're upto date.
- **Pydantic config = validated, frozen submodels; runtime state = plain dataclass.** Mixing the two tempts you to validate state on every write and erodes the boundary. Keep Pydantic for config (loaded once at startup, immutable after), plain dataclass for `WorkspaceState` (mutable, persisted to JSON).
- **`${repo}` / `${repo_name}` placeholders are expanded at consume time, not validate time.** A single global config must serve every repo without re-validation; deferring expansion to `WorkspaceManager.create()` is what makes that work.
- **`agents` lists merge by `name` across cascade layers; other lists replace wholesale.** Codified in `_merge_agents` in `grove/core/config.py`. If you add another list-of-named-things, decide explicitly which behavior it needs and document it next to the field.
- **Grove never runs `git commit` or `git push`.** Pause refuses dirty worktrees with a typed error so the user sees uncommitted work; we do not auto-commit on their behalf. The user owns commits/pushes via the dedicated tmux shell window or lazygit.
- **Side effects only in `grove/core/git.py` and `grove/core/tmux.py`.** Manager sequences them; everything else is pure. When a new I/O concern shows up, route it through one of those two files or create a third side-effect module ŌĆö do not scatter `subprocess.run` calls.
- **Test seam = the public module surface.** `tests/conftest.py` monkey-patches `grove.core.tmux.create_session` etc. directly ŌĆö no Protocol abstraction. If you rename a public function, the test breaks loudly, which is the point.
- **Cross-platform: pin `mypy<2.0` until 2.0 ships Linux x86_64 wheels.** Use `pathlib.Path` everywhere, `os.replace` for atomic writes, `shell=False` with list args for every subprocess call. tmux on Windows requires WSL2 ŌĆö surface a clear `TmuxError` instead of half-supporting Windows-native.
- **CI is Linux-only (`ci.yml` runs `ubuntu-latest`, no OS matrix).** Grove is a tmux-centric terminal app plus a loopback daemon; tmux is a hard runtime dependency (WSL2 on Windows) and native packaging was retired with the desktop app, so the old `[ubuntu, macos, windows]` matrix burned minutes for no signal ŌĆö the Windows leg only ever failed on tmux-absent corners. The Windows/macOS-specific *code* (POSIX import guards, `os.replace`, `platformdirs` patching, the `shutil.which("bash")` probe) and the lessons documenting it stay: they're cheap defensive correctness, and dropping CI coverage is not a license to write Linux-only code. It does mean **those defenses are now unverified by CI** ŌĆö a Windows/macOS regression won't be caught automatically, so reason about them by hand when you touch path/subprocess/POSIX-module code. `make lint` runs `ruff format --check` (not just `ruff check`) ŌĆö a `ruff check --fix` that reflows code can still leave a format-check failure, so always run the full `make lint` before pushing, never just `ruff check`.
- **Loguru is configured once in the Typer callback.** Per-subcommand `_configure_logging()` calls leak the default DEBUG handler when a subcommand is invoked before any other code runs. The `@app.callback(invoke_without_command=True)` runs for every invocation including subcommands; that's the right home for it.
- **Keep `import-linter` honest with `include_external_packages = true`.** Without it, contracts that forbid third-party modules (textual, rich, typer, click) silently pass.
- **`peek()` is best-effort by contract ŌĆö it never raises.** Helpers can fail (git binary missing, tmux gone, branch deleted); peek returns zeros / empty rather than break the render loop. Lifecycle methods (`create`, `pause`, `kill`) keep the loud failure surface ŌĆö those are transactional.
- **Init outcome is captured once, in one helper (`_init_outcome` in `manager.py`).** Both `create()` and `resume()` apply it via `_replace(state, **_init_outcome(...))`. Don't reimplement the OK/FAILED/SKIPPED branching at the call site ŌĆö the helper exists so the two paths can't drift.
- **Don't commit `docs/superpowers/specs/...`.** Those are session-local working memory used by the brainstorming ŌåÆ writing-plans ŌåÆ implementation skills, not durable docs the team consumes. The repo's `.gitignore` excludes them; if a spec ever ends up tracked, untrack it. Durable lessons belong in CLAUDE.md (per the routing table), not in spec docs that go stale.
- **Test assertions that compare paths from `git` CLI output to Python `str(Path)` need explicit normalization on Windows.** Git emits `/`, Python emits `\`. Linux makes the asymmetry invisible ŌĆö both produce identical strings ŌĆö so the bug only surfaces on Windows CI. Compare resolved `Path` objects (`Path(p).resolve() in {Path(q).resolve() for q in git_out}`), never raw strings. The same rule applies to any cross-tool path comparison (lazygit, gh, etc.).
- **`shutil.which("bash")` on Windows GitHub runners is misleading.** `bash.exe` resolves to the WSL launcher. With no installed distribution it prints "Windows Subsystem for Linux has no installed distributions." and exits 1 for *every* `bash -c ŌĆ”`. To gate tests that actually exec a shell, run a probe (`bash -c "echo grove-shell-probe"`) and verify the output ŌĆö `which` alone passes the stub. Same gotcha applies to `git` only by accident (Windows runners include real Git).
- **Pane-target resolution is a manager-side policy, never a tmux-side one.** `tmux.list_windows(session)` is mechanism ŌĆö list whatever the session has, never raise. `WorkspaceManager.pane_target(workspace_id)` is policy ŌĆö prefer the configured `agent_window_name`, fall back to the first non-`shell` window, fall back to `shell`, return `None` only when the session reports no windows. `_capture_pane` (used by `peek` and `peek_pane`) consumes it. The fallback exists because workspaces created or reorganized outside Grove (or whose agent window auto-closed when the agent process exited) have arbitrary window layouts, and the rail must show *something* live whenever the session has any output. Hard-coding `f"{session}:{agent}"` was the original peek-empty bug ŌĆö don't reintroduce it.
- **`WorkspaceStatus` is one enum with two domains: persistent intents + computed views.** Persisted to JSON: `RUNNING` / `PAUSED` / `ERROR` (the things `pause()` / `resume()` / `respawn()` / `_record_error` write). Computed at read time and never persisted: `ACTIVE` / `IDLE` / `OFFLINE` / `ORPHANED`. `WorkspaceManager._reconcile_status` is the **single** policy site that promotes a persisted intent ŌåÆ displayed status, applied by `list()` / `peek()` / `peek_pane()` / `attach()` / `respawn()`. `JsonWorkspaceStore.save` rejects writes whose status is not in `PERSISTED_STATUSES` (defense in depth). Legacy on-disk values (`stale` from before this split) coerce via `_decode_status` to the persisted intent that produced them. Adding a new computed status: (1) extend the enum, (2) add to `STATUS_HEX` + `STATUS_GLYPH` + `STATUS_LABEL`, (3) extend `_reconcile_status`, (4) extend `_AVAILABLE_KEYS_BY_STATUS` in `screens/list.py` to gate the footer. Do not branch on raw `RUNNING` outside `_reconcile_status` and the lifecycle methods that *write* it ŌĆö every consumer should see the promoted value.
- **Active vs Idle = `tmux #{window_activity}`, NOT `#{pane_activity}`.** tmux's `pane_activity` format variable was added in tmux Ōēź3.4 and **returns an empty string on tmux Ōēż3.3** (verified on 3.2a, the version Ubuntu 22.04 ships). An empty value lands in `pane_activity_seconds_ago`'s `not raw` branch ŌåÆ `None` ŌåÆ reconciliation treats every workspace as IDLE on those systems, which is wrong-by-omission rather than wrong-by-action but still useless as a signal. `#{window_activity}` is broadly supported and updates whenever any pane in the window emits output. Grove's layout is one pane per window (shell, agent), so window-level == pane-level for our purposes. Public helper name stays `pane_activity_seconds_ago` (it still measures "the agent pane's activity"); the implementation reads `window_activity`. Threshold is `cfg.tmux.activity_threshold_seconds` (default 5s). Do **not** introduce a parallel "captured snapshot diff" cache ŌĆö tmux is the source of truth, one subprocess per running workspace per slow tick. Future timestamps and non-numeric output coerce to `None`; reconciliation treats unknown age as IDLE (fallback, not signal). Probe with `tmux display-message -p -t <target> -F '#{window_activity}'` ŌĆö if it returns an integer, you have a usable signal; if empty or non-digit, the reconciler will degrade to IDLE for that workspace.
- **`BranchPlan` is a Pydantic discriminated union, not a dataclass ŌĆö and that rule generalizes.** Branch-source intent (Auto / NewNamed / ExistingLocal / TrackRemote) is a *public contract*: the TUI constructs a variant today, a future API server will receive it as JSON, a future web client will build it from form fields. That's the boundary every Pydantic-vs-dataclass call hinges on. Each variant carries a `kind: Literal["..."]` tag plus its own fields, and the union is `Annotated[Union, Field(discriminator="kind")]` ŌĆö Pydantic v2 dispatches by tag and generates JSON Schema for the whole tree. Inside the engine, each variant has a `resolve(cfg, title, ts) ŌåÆ ResolvedBranch` method; `ResolvedBranch` is a plain frozen dataclass because it never crosses a wire. Same shape for any future "what does the user want" intent (e.g. attach plans, rename plans): Pydantic outward, dataclass inward.
- **Pydantic v2's default regex engine (Rust) has no lookahead / lookbehind.** The natural `^(?!-)[A-Za-z0-9._/\-]+$` for "no leading dash" raises `SchemaError: look-around ... is not supported` at class-construction time. Workaround: split the character class ŌĆö `^[A-Za-z0-9._/][A-Za-z0-9._/\-]*$` matches the same set without lookahead. Same gotcha for any `(?=ŌĆ”)` / `(?<ŌĆ”)` / non-greedy edge cases. If a regex truly needs lookaround, switch the field's regex engine via `Field(..., pattern_engine="python-re")` (Pydantic v2.10+).
- **`branch_provenance` defaults to `GROVE_CREATED` so legacy state files load without migration.** The dataclass field has a default, so `WorkspaceState(**data)` for an old JSON record (no `branch_provenance` key) picks up the default. Historical behavior was "Grove always created the branch" ŌĆö the default matches it exactly. New stores write the field explicitly; old reads coerce missing ŌåÆ GROVE_CREATED. No version bump needed for a JSON-cosmetic addition.
- **The `~/.local/bin/grove` shim that systemd / shell PATH resolves is whichever `grove` was last installed via `uv tool install`.** A vanilla `uv tool install grove` (or the install.sh from main) pulls the published PyPI wheel ŌĆö which is the **release version**, not the local checkout. Symptom: a freshly-added daemon endpoint or engine method shows up green in `pytest` but the running daemon serves 404 / `No such command`. Fix is `uv tool install --reinstall --force --editable '.[daemon]'` from the repo root, then `systemctl --user restart grove-daemon` (or whatever process owns the binary). The `[daemon]` extra is load-bearing ŌĆö without it `grove daemon` errors with `No such command` because the Typer mount is gated by `try: from grove.daemon.cli import app`. When in doubt: `which grove`, `readlink -f $(which grove)`, and check the version via the daemon's `/openapi.json` (compare its route list to your branch's `tests/daemon/test_app.py`). Editable install is also the right default for any host running grove out of a worktree ŌĆö code edits land instantly with just a service restart.
- **systemd-user units ship as `*.service.in` templates rendered by Makefile sed substitution.** [`packaging/systemd/`](packaging/systemd/) holds the templates; root `Makefile` carries `systemd`, `systemd-enable`, `systemd-disable`, `systemd-uninstall`, `systemd-status`, `systemd-print` targets. Webapp install is **opt-in** via `WITH_WEBAPP=1`; default is daemon-only because most users don't run the dashboard. The webapp unit `Wants=` the daemon (not `Requires=`) ŌĆö daemon failure doesn't tear the webapp down, the in-app status bar surfaces "unreachable" instead. Use `Wants=` for any future companion service that polls or talks to the daemon. Ports / hosts / paths flow through Make variables (`DAEMON_PORT`, `WEBAPP_HOST`, `WEBAPP_DIR`, `GROVE_BIN`, `NPM_BIN`); auto-detected via `command -v` at install time. Tests: `tests/test_systemd_packaging.py` invokes `make systemd-print` and asserts placeholder substitution + the `Wants=` invariant ŌĆö never write to `~/.config/systemd/user` from tests (use the print/dry-run target). When adding a new service unit, follow the same shape: ship a `.in` template, add a `_systemd-install-<name>` recipe, gate inclusion in `systemd` via a Make-level `$(if $(WITH_<NAME>),...)` conditional ŌĆö never via shell `if`/`fi` inside a recipe (multi-line shell heredocs inside Make `define` blocks fail in subtle ways).
- **`kill()` is the only lifecycle op that ever deletes a branch, and it never touches a remote.** `pause` removes the worktree, keeps the branch (worktree management is Grove's domain; the branch is the user's). `resume` recreates the worktree from the persisted branch. `respawn` only touches tmux. `kill(workspace_id, *, delete_branch=None)` is the sole branch-deleting path: when `delete_branch` is unspecified, Grove resolves it from `state.branch_provenance` (GROVE_CREATED ŌåÆ True, USER_ATTACHED ŌåÆ False); the user can override either way in the TUI's kill modal. **Remote branches are never touched, period.** There is no flag to opt into remote deletion ŌĆö that's `git push --delete` territory and stays in the user's shell with their own credentials. The kill event detail emits `branch_deleted: "true" | "false"` so clients can flash the right message.
- **FastAPI request-body Pydantic models live at module scope, never in a closure.** Under `from __future__ import annotations` (which we use everywhere) handler signatures become string forward-refs. Pydantic v2's `TypeAdapter` rebuild walks the function's globals to resolve those strings ŌĆö it does **not** see closure scopes. A `class _Body(BaseModel)` defined inside `build_app()` and referenced as `body: _Body` in a handler will raise `PydanticUserError: TypeAdapter[_Body] is not fully defined` on the first request. Module-top with a leading underscore is the right placement: the underscore preserves the daemon-internal contract (CLAUDE.md "public surface small") while the module scope makes the symbol visible to the resolver. The same gotcha applies to any `Annotated[Union, Field(discriminator=...)]` you construct inline ŌĆö keep wire types module-top, dataclass IR can stay scoped.
- **`platformdirs` on Windows ignores `APPDATA` / `LOCALAPPDATA` env vars when `ctypes` is available.** Recent platformdirs (Ōēź4.x) defaults to resolving the user config dir via `SHGetKnownFolderPath` through `ctypes` on Windows; the env-var path is only the fallback when ctypes can't be imported. So `monkeypatch.setenv("LOCALAPPDATA", str(tmp_path))` does nothing on Windows CI runners ŌĆö the real user dir is still consulted, the test write lands outside `tmp_path`, and `rglob` returns empty. The right sandbox is `monkeypatch.setattr("grove.core.paths.user_schema_path", lambda: tmp_path / "...")` ŌĆö patches the Grove function directly, deterministic on every OS, and matches the pattern `tests/conftest.py::tmp_state_dir` already uses for `user_state_path` / `user_config_path` / `user_schema_path`. Same gotcha for any test that needs to redirect `~/.grove`, `~/.config/grove`, or any `platformdirs.user_*_dir`-derived path on Windows.
- **POSIX-only stdlib modules (`pty` / `fcntl` / `termios`) MUST be guarded with `if sys.platform != "win32":`.** A bare top-level `import fcntl` raises `ModuleNotFoundError` on Windows during *test collection*, which fails every test that transitively imports the module ŌĆö Linux/macOS green, Windows red across the board with no useful failure context. The guard pattern: import the POSIX modules conditionally, then have the consuming class raise `NotImplementedError("requires POSIX (use WSL on Windows)")` from its first runtime entry point. The module loads cleanly on every platform; the unsupported path fails loudly only when actually exercised. Same approach as the existing `tmux on Windows requires WSL2` lesson ŌĆö degrade to a typed error, never to an import failure. Pinned by `tests/client/test_local_attach.py` carrying `pytestmark = pytest.mark.skipif(sys.platform == "win32", ...)` so the suite stays runnable on Windows.
- **Daemon listens on `127.0.0.1` only; remote access is via SSH port-forward ŌĆö auth is deferred precisely because the loopback constraint is load-bearing.** `grove daemon serve` binds loopback by default and there is no `--host 0.0.0.0` blessing. `SshTransport` reuses one `asyncssh.SSHClientConnection` for both the HTTP forward (`forward_local_port`) AND interactive `tmux attach` (`SSHClientProcess`) ŌĆö one TCP session, two jobs. Adding a network-exposed listener would force every endpoint into a real auth design (token? mTLS?) ŌĆö instead we trust the user's existing SSH credentials. If you ever feel tempted to widen the bind, the right answer is "no": stand up an additional layer (Tailscale, WireGuard, an authenticated reverse proxy) outside Grove, not inside.
- **`RepoRegistry` caches one `WorkspaceManager` per `Path.resolve()`d repo_root.** The daemon serves multiple repos out of one process; the registry is the cache that prevents `WorkspaceManager.__init__` from running per request. Symlinked paths collapse to a single Manager (canonicalize via `.resolve()`). `known_roots()` is fresh-from-store every call so newly created repos appear without restart. The cache is unbounded by design ŌĆö V1 is loopback-only with small N ŌĆö and `/workspaces` walks `known_roots()` then dispatches to each Manager (N+1 store reads is real but dwarfed by the per-workspace tmux-subprocess reconcile cost).
- **`AttachSession` has two impls; both share one Protocol so a client's terminal bridge is transport-agnostic.** `LocalAttach` uses stdlib `pty.fork` + `asyncio.add_reader` for local backends (one PTY pair per attach, lifecycle on close: SIGHUP ŌåÆ waitpid ŌåÆ close). `SshAttach` uses `asyncssh.create_process` over the existing `SSHClientConnection` for remote backends (binary I/O via `encoding=None`). Production callers always run `tmux attach -t <session>`; tests use `_command_override` (kwarg-only, underscore-prefixed) to substitute deterministic commands like `cat` or `echo`. Once the session is attached, **tmux owns the UI** ŌĆö Ctrl-b w / Ctrl-b s / copy mode / status line ŌĆö Grove never reimplements a terminal multiplexer.
- **Wire boundary = Pydantic Views; engine state = plain dataclass. Never mix the two.** `WorkspaceStateView` / `WorkspacePeekView` / `AttachInstructionView` / `CommitSummaryView` live in `grove/core/contracts/views.py` and exist solely to serialize engine dataclasses across HTTP. Internal-only fields (`init_log_path`, `init_env`) deliberately do NOT appear on the wire. Each View has a `from_*` classmethod that adapts the dataclass; `frozen=True` on the model_config catches accidental mutation. New endpoint? Add a View if the response shape isn't already covered ŌĆö never expose the dataclass directly. The View boundary is the single coupling point between engine and clients (TUI, daemon, future web).
- **Daemon error envelope is `{"detail": {"error": <code>, "message": <text>}}`, mapped via `_grove_error_to_http`'s `code_map`.** Subclass entries MUST precede the parent class ŌĆö the loop is a linear `isinstance` scan, so the first match wins. `BranchError` is the catch-all for any new subclass we forget to enumerate; `WorkspaceNotFound` and `WorkspaceStateError` map to 404 / 409 with typed codes. Bare `GroveError` falls through to 500 / `grove_error` ŌĆö not great for "unknown agent" (manager.py raises bare GroveError there) but acceptable for V1; widen the map when a new typed subclass shows up. Clients (`GroveClient._raise_for_status`) parse the envelope and raise `ProtocolError(code=..., message=..., status=...)`, preserving the typed error all the way to the UI.
- **`grove.cli` adopts the TUI's Typer app (not `add_typer`-mounts it) to preserve back-compat.** Replacing `[project.scripts] grove = "grove.tui.cli:app"` with `grove = "grove.cli:app"` would have buried every existing TUI command under a `grove tui` namespace if we used `add_typer(tui_app, name="tui")`. Instead `grove.cli` IS the TUI app, then `add_typer`s the daemon subcommand with a name. `grove ls` and `grove version` still work; `grove daemon serve` mounts on top. The daemon is guarded by `try: ... except ImportError`, so a bare `pipx install grove` install still has a working CLI without its extras.
- **Title is identity-seed-only at create; rename is metadata-only.** `WorkspaceManager.create()` derives the worktree path and tmux session name from `slug(title) + ts` at the moment of creation; both are persisted strings on `WorkspaceState` after that. `WorkspaceManager.update(workspace_id, *, title=..., description=...)` is the rename path: it changes only `title`, `description`, and `updated_at` ŌĆö the worktree directory on disk and the live tmux session keep their original names forever, because renaming them would break attached clients and the user's muscle memory. Same shape for the optional `description` field: it never feeds into anything path-shaped, just rendered text. The engine uses an `_UNSET` sentinel on the public method signature so callers can distinguish "leave alone" (default) from "set to None" (clear description); `None` means "clear" for description, `None`/`_UNSET` means "leave alone" for title (which can never be cleared because workspaces require a non-empty title at all times). Wire-side, `UpdateWorkspaceRequest` uses Pydantic's idiomatic JSON-PATCH convention: `null`/omitted = "do not change", and to clear description the client sends `""`. The daemon translates wire `null` ŌåÆ not passing the kwarg. Refuses an empty body with 422 (no-op PATCH would bump `updated_at` for free); refuses ORPHANED with 409 (record headed for kill ŌĆö rename would add confusion). The matching event kind is `"updated"` with `title_changed`/`description_changed` flags so subscribers know what shifted without diffing.
- **Recovery from session-vanished is `respawn`, not `resume`.** `pause` removes the worktree; `resume` recreates it from the branch. If a tmux session vanishes externally while the worktree is intact, neither matches. The dedicated `respawn(workspace_id)` lifecycle: persisted status must be OFFLINE (`ensure_can_respawn`), worktree dir must exist on disk (else ORPHANED, kill-only), then `tmux.create_session` + `build_workspace_layout` and the persisted intent stays RUNNING. Init script does not re-run by default; `cfg.init_script.run_on_resume` opts in for parity with `resume`. Bound to key `o` (for "online"). Never overload `resume` to handle both ŌĆö the verb means a different thing on each path and `ensure_can_resume` would have to widen its contract.
- **Published docs have two audiences; keep them physically separated.** The mkdocs site (`docs/`, deployed to the Pages site) is *product/user* documentation, split in its own nav into **User Guide** and **Developer Reference**. Operational/contributor detail for a component ŌĆö the dashboard dev server (`npm run dev`), wire-type codegen, the test pyramid, BFF route internals ŌĆö lives in that component's README (e.g. `webapp/README.md`), **not** in the site's User Guide. User-facing webapp / auth / daemon docs live under `docs/` *Use* (`use-webapp.md`, `use-auth.md`, plus the `grove daemon` / `grove auth` blocks in `use-cli.md`); the site links *out* to the README for contributor work rather than absorbing it, and the README banners *back* to the site for product usage. When a feature spans both audiences, write each half in its own surface and cross-link ŌĆö never blur product and contributor prose on one page. The `--print-port`-style internal flags get a user-facing gloss on the site (what it does), with the implementation rationale (LocalTransport auto-discovery) staying in code comments. This is the doc analogue of the engine's "public surface small, internals private" rule.

- **Docs presentation and writing style for the published site (learned from peer products like superconductor.com and direct user feedback).** Presentation: a skimming reader reads the headline, glances at one product screenshot, then scans section headers; they do not read paragraphs first. So **real product screenshots are signal and carry the message; decorative chrome (icon-tile galleries, status-chip walls, pill rows, numbered-lifecycle strips) is noise ŌĆö cut it.** The landing page sells and routes; depth lives in the User Guide. Erring toward "comprehensive" on the home page reads as bloat, and restraint is the quality signal.
