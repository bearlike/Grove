# grove.core — engine: lifecycle, config, side effects, status, registry, activity

> ↑ [root](../../../CLAUDE.md)

The engine: the manager sequences worktree/tmux lifecycle, the config cascade resolves user policy, status reconciliation promotes intents to displayed states, and the registry + activity service aggregate across repos. All side effects funnel through two files; everything else is pure.

Children: [contracts](contracts/CLAUDE.md) (wire shapes) · [agents](agents/CLAUDE.md) (agent adapters).

## Lifecycle & manager

- **Init outcome is captured in one helper (`_init_outcome` in `manager.py`).** `create()` and `resume()` both apply it via `_replace(state, **_init_outcome(...))`; never reimplement the OK/FAILED/SKIPPED branching at a call site, so the paths can't drift.
- **`kill()` is the only op that deletes a branch, and it never touches a remote.** `pause` removes the worktree, keeps the branch (worktree is Grove's, branch is the user's); `resume` recreates the worktree from the branch; `respawn` touches only tmux. `kill(id, *, delete_branch=None)` resolves an unspecified flag from `state.branch_provenance` (GROVE_CREATED → True, USER_ATTACHED → False); the TUI modal can override. Remote branches are never touched — `git push --delete` stays in the user's shell. Kill event emits `branch_deleted: "true"|"false"`.
- **Recovery from a vanished tmux session is `respawn`, not `resume`.** `respawn(id)` requires persisted OFFLINE (`ensure_can_respawn`) and the worktree dir on disk (else ORPHANED, kill-only), then recreates session + layout; intent stays RUNNING. Init re-runs only with `cfg.init_script.run_on_resume`. Never overload `resume` — the verb means different things per path.
- **Title is identity-seed-only at create; rename is metadata-only.** `create()` derives the worktree path and tmux session name from `slug(title) + ts` once, then persists them. `update(id, *, title, description)` changes only `title`/`description`/`updated_at` — dir and session keep their names forever (renaming breaks attached clients). Uses an `_UNSET` sentinel: `None` = clear (description only), `_UNSET` = leave alone; title can never be cleared.

## Placement (worktree vs root)

- **Placement gates every worktree side effect and is safety-critical.** A workspace is WORKTREE (dedicated dir under `${repo}/.worktrees`) or ROOT (runs in `repo_root`, adopts live HEAD, Grove creates nothing). Modeled as the `RootBranch` discriminated-union variant in [contracts](contracts/CLAUDE.md) — not a boolean — so illegal combos are unrepresentable. `RootBranch.resolve()` returns `placement=ROOT`, `provenance=USER_ATTACHED`, `name=""`; the empty name is the sentinel the manager fills from `git.current_branch()` (detached HEAD → `"HEAD"`).
- **The `Placement` enum lives in `workspace.py`, NOT `contracts/branch_plan.py`** — `branch_plan` imports from `workspace`, so the enum sits on the depended-upon side or the import cycles. `WorkspaceState.placement` defaults to WORKTREE so legacy JSON loads with no migration.
- **The gate is one check (`placement is ROOT`) at each worktree side-effect site:** `create` skips `worktree_add` + branch creation; `kill` skips `worktree_remove` and **forces `delete_branch=False` even when the caller passes `True`**; `_rollback_create` reuses the gate so a failed root create can't destroy the repo; `respawn` never re-runs init for root. Root lifecycle is **create/kill/respawn only** — `ensure_can_pause`/`ensure_can_resume` refuse it, and reconciliation never yields ORPHANED. Tests must spy that `worktree_remove`/`branch_delete` are never *called* for root — git refuses to remove the main worktree, so a survival-only assertion passes even with the gate missing.

## Config cascade

Six-layer cascade philosophy (mechanism, not policy) lives in [root](../../../CLAUDE.md).

- **Pydantic config = validated, frozen submodels; runtime state = plain dataclass.** Config loads once and is immutable; `WorkspaceState` is mutable, persisted to JSON. Mixing them tempts per-write validation and erodes the boundary.
- **`${repo}` / `${repo_name}` placeholders expand at consume time, not validate time.** One global config serves every repo without re-validation; expansion deferred to `create()` is what makes that work.
- **`agents` lists merge by `name` field-by-field; other lists replace wholesale.** Codified in `_merge_agents` (`config.py`): an overlay entry refines the base (`{**base, **overlay}`), so overriding just the `claude` agent's `command` keeps its `kind="claude_code"`. Any new list-of-named-things: decide merge vs replace explicitly, document it at the field.
- **`skip_init` is a per-create override of `init_script.enabled` — create-scoped, never persisted.** `create()` runs init only when `init_script.enabled and not request.skip_init`, else `InitStatus.SKIPPED`. Gating at the call site lets one create opt out without touching config. Not stored, never re-applied on resume/respawn (those keep their own `run_on_resume` gate). Placement and skip-init are independent in the engine.

## Side effects & status

- **Side effects only in `grove/core/git.py` and `grove/core/tmux.py`.** Manager sequences them; everything else is pure. A new I/O concern routes through one of those (or a new dedicated side-effect module) — never scatter `subprocess.run`.
- **Grove never runs `git commit` or `git push`.** Pause refuses dirty worktrees with a typed error; we never auto-commit. The user owns commits/pushes via the tmux shell window or lazygit.
- **`peek()` is best-effort by contract — it never raises.** Helpers can fail (git/tmux gone, branch deleted); peek returns zeros/empty rather than break the render loop. Lifecycle methods keep the loud, transactional failure surface.
- **Pane-target resolution is manager-side policy, never tmux-side.** `tmux.list_windows(session)` is mechanism — list whatever exists, never raise. `pane_target(id)` is policy — prefer configured `agent_window_name`, fall back to first non-`shell` window, then `shell`, return `None` only with no windows. Hard-coding `f"{session}:{agent}"` was the original peek-empty bug.
- **`WorkspaceStatus` is one enum, two domains.** Persisted: `RUNNING`/`PAUSED`/`ERROR` (written by `pause`/`resume`/`respawn`/`_record_error`). Computed at read time, never persisted: `ACTIVE`/`IDLE`/`OFFLINE`/`ORPHANED`. `_reconcile_status` is the **single** policy site promoting persisted → displayed; `JsonWorkspaceStore.save` rejects non-`PERSISTED_STATUSES` (defense in depth). Adding a computed status: extend the enum + `STATUS_HEX`/`STATUS_GLYPH`/`STATUS_LABEL` + `_reconcile_status` + the TUI footer gate. Never branch on raw `RUNNING` outside `_reconcile_status` and the lifecycle writers.
- **Active vs Idle = `tmux #{window_activity}`, NOT `#{pane_activity}`.** `pane_activity` (tmux ≥3.4) returns empty on tmux ≤3.3 (Ubuntu 22.04 ships 3.2a) → every workspace reads IDLE. `#{window_activity}` is broadly supported and, since Grove runs one pane per window, equivalent. Public helper stays `pane_activity_seconds_ago`; implementation reads `window_activity`. Threshold `cfg.tmux.activity_threshold_seconds` (5s). No snapshot-diff cache — tmux is source of truth, one subprocess per running workspace per slow tick. Unknown/future/non-numeric age → `None` → IDLE.
- **`branch_provenance` defaults to `GROVE_CREATED` so legacy state files load without migration.** Default matches historical behavior; new stores write it explicitly, old reads coerce missing → GROVE_CREATED. No version bump — same precedent as `placement` and `agent_session_id`.

## Registry & activity

- **`RepoRegistry` (`grove/core/registry.py`) caches one `WorkspaceManager` per `Path.resolve()`d repo_root.** Pure engine (cfg + store + manager cache), consumed by daemon, `ActivityService`, and TUI dashboard; `grove.daemon.repos` re-exports for back-compat. Symlinks collapse to one Manager via `.resolve()`. `known_roots()` is fresh-from-store every call so new repos appear without restart. Unbounded by design (loopback-only, small N).
- **`ActivityService` (`grove/core/activity.py`) is the cross-project aggregation hub: one engine source, two renderers.** Enumerates every workspace across every repo via `RepoRegistry`, resolves each session through the adapters, exposes `snapshot()` + `subscribe()`/`poll_once()`.
- **`_blend` is the single status-blend policy site** (the `_reconcile_status` analogue). Reuses the manager's already-reconciled `WorkspaceStatus` for the tmux dimension (no second tmux call): transcript WAITING/ERROR/UNKNOWN pass through; transcript WORKING + workspace ACTIVE → WORKING, else IDLE; no transcript yet → STARTING.
- **`AgentActivityState` is a separate axis from `WorkspaceStatus` — never overload the workspace enum.** Agent axis and tmux axis are independent; a card renders both.
- **The timer lives at the edge.** Daemon lifespan / TUI ticker call `poll_once()`; the service keeps time-of-day out of its core. Engine emits `DashboardDelta` (dataclass) on its bus; the wire `DashboardEvent` + `*View` mirrors live in [contracts](contracts/CLAUDE.md) `activity.py`, importing engine dataclasses under `TYPE_CHECKING` only so a contracts import never drags in manager/registry.
- **A card carries three deliberately-separated signals; "latest activity" is the last commit, not the transcript.** *Ongoing action* (live, ephemeral) is the blended transcript state (with tmux fallback for sessionless workspaces — done client-side off `WorkspaceStatus`); *latest activity* (durable, timestamped) is `recent_commits[0]` on the workspace branch; *commit history* is the rest. The transcript `current_task` is kept ONLY as the live ongoing-action line — commits are the precise record of *what was done, and when*, where the transcript task text goes stale. `WorkspaceActivity` carries `recent_commits` (one cheap `git log -3` per row per tick, best-effort like `ahead_behind`/`diff_stats` — never breaks the snapshot) and `observed_at` (the per-card "updated" time). Two fingerprint invariants: **`observed_at` is excluded from `fingerprint`** (it changes every tick → would re-emit every card every poll), and the **last commit sha IS in the fingerprint** so a fresh commit streams promptly even when it doesn't move ahead/behind (an amend on the tip). `dirty_files` (uncommitted-path count, untracked included) is also in the fingerprint — the agent-is-editing signal must stream *before* anything is committed; it costs one more best-effort git call per row per tick and reads 0 for a paused workspace's missing dir.
- **`WorkspaceState.agent_kind` is persisted at create so the dashboard resolves an adapter without re-reading config — which makes an agent scopable to a repo.** The daemon's `ActivityService` runs off ONE global config (`load_config(repo_root=None)`; per-repo project config is deliberately not loaded on the read path, or it pulls each project's defaults over the daemon's own `auth`/`daemon` sections). Agent lists merge additively (union by `name`, no per-repo exclude), so the only way to keep an agent out of other repos' menus is to define it solely in those repos' `.grove/config.json` — but then `find_agent(state.agent_name)` returns `None` → `generic` → blank agent axis. Fix: `create()` captures the resolved `AgentSpec.kind` into `agent_kind` (the create-time-fact pattern); `sessions_for` **prefers `state.agent_kind`**, config lookup only for legacy `None`. `AgentKind = Literal["claude_code","generic"]` lives in `config.py`, imported by `workspace.py` (no cycle). Caveat: daemon-side lifecycle ops (`/respawn`) still resolve the agent *command* from global config, so they can't relaunch a repo-scoped agent — the TUI is the lifecycle surface for those.
- **A workspace with no minted `agent_session_id` still surfaces its live session via discovery — `sessions_for` (public seam since the TUI list tick became its second consumer alongside the daemon poll) must NOT early-return on a falsy id.** Three shapes have no id: a non-`claude_code` agent at create, a pre-minting record, a hand-started `claude`. The old `if not id: return []` blanked the entire agent axis though a transcript was on disk. Invariant: with no minted id, recover via `adapter.discover_sessions(worktree, exclude_id=None)` and adopt the single most-recent transcript as the `fs_discovered` primary. **Adapter-gated** (generic/shell → no-op) and **NOT gated by `cfg.hooks.enabled`** (discovery is a read-only fs glob, always runs; `hooks.enabled` gates only sidecar install + augmentation of an already-minted session). Discovery mechanics (newest-first-by-mtime, cwd-preamble scan, `CLAUDE_CONFIG_DIR` cascade) live in [agents](agents/CLAUDE.md). Operational corollary: a custom-named Claude agent must declare `kind: "claude_code"` (merge-by-name doesn't inherit the built-in `"claude"`'s kind), and a profile with a custom `CLAUDE_CONFIG_DIR` needs that dir on the daemon process's own env or its sessions are invisible.

## Session exploration (`sessions.py` + `grove sessions`, issue #26)

- **`SessionExplorer` answers one question — "which agent sessions belong to this project, and how do I read one?" — and is read-only by construction.** It composes the adapters' scans (`list_sessions` per cwd, `locate_transcripts`/`read_turns` per session); it never launches, mutates, or deletes. A new agent tool needs zero changes here — `all_adapters()` picks its adapter up.
- **`from_cwd` binds the manager to the MAIN worktree root (`GitRepo.worktree_paths()[0]`), never to `detect_root(cwd)` directly.** The workspace store is keyed by the main root; binding to a linked worktree's own root finds zero workspaces. `worktree_paths()` works from inside any worktree (git reports the whole family) and returns main-first.
- **The scan set is the union of live `git worktree list` AND every workspace's persisted `worktree_path`.** The first covers hand-made worktrees Grove never managed; the second covers paused workspaces whose directory is gone — transcripts outlive worktrees (discovery scans the encoded-cwd projects folder, not the worktree itself).
- **Provenance is minted-id equality, workspace association is cwd equality — in that order.** A session whose id matches a workspace's `agent_session_id` is `grove_launched` and binds to that workspace even if found elsewhere; otherwise the workspace owning the scanned cwd annotates it (`fs_discovered`).
- The `grove sessions` CLI (`tui/cli_sessions.py`: `list` with `--agent/--workspace/--since/--limit/--json`, `show --last N`, `dump --jsonl`) is a thin renderer over the explorer — filters and resolution (unique id prefix) live engine-side so the daemon/webapp can reuse them later.
- **`for_workspace(id)` / `turns_for(listing)` are the bounded per-request variants the daemon serves.** `list()` full-parses every transcript across every worktree — fine per CLI invocation, too heavy per HTTP request; `for_workspace` scans exactly one cwd, `turns_for` reads from an already-resolved listing (skipping `resolve()`'s full scan). Inside this class body a `list[...]` return annotation resolves to the `list` *method*, not the builtin — return tuples (the same shadowing trap documented for `WorkspaceManager`).

## CLI

- **`grove.cli` adopts the TUI's Typer app (does not `add_typer`-mount it) to preserve back-compat.** Mounting under a `tui` namespace would bury `grove ls` / `grove version`. Instead `grove.cli` IS the TUI app, then `add_typer`s the daemon subcommand. The daemon import is guarded (`try: ... except ImportError`), so a bare `pipx install grove` still has a working CLI.
- **Loguru is configured once in the Typer callback.** Per-subcommand `_configure_logging()` leaks the default DEBUG handler. `@app.callback(invoke_without_command=True)` runs for every invocation including subcommands — the right home.

## Cross-platform

- **Pin `mypy<2.0` until 2.0 ships Linux x86_64 wheels.** Use `pathlib.Path` everywhere, `os.replace` for atomic writes, `shell=False` with list args for every subprocess. tmux on Windows requires WSL2 — surface a clear `TmuxError`, never half-support Windows-native. Degrade unsupported platforms to a typed error, never an import failure (same for POSIX-only stdlib modules — guard them behind `if sys.platform != "win32":`).
